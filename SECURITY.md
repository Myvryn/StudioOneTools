# Security Policy

Thank you for taking the time to help improve the security of **StudioOneTools**.  
We take security seriously and appreciate responsible disclosures.

---

## 🔒 Supported Versions

StudioOneTools is maintained on a best‑effort basis.  
Security fixes will be applied to the **latest release** and **main branch**.

Older versions may not receive patches.

---

## 🐞 Reporting a Vulnerability

If you believe you have found a security vulnerability, **please do not open a public issue**.

Instead, contact the maintainer directly:

**security contact:**  
your-email@example.com  
(Replace this with your preferred email)

Please include:

- A detailed description of the vulnerability  
- Steps to reproduce  
- Potential impact  
- Any suggested fixes (optional)

You will receive an acknowledgment within **72 hours**, and a full response within **7 days**.

---

## ✅ Responsible Disclosure Expectations

When reporting a vulnerability:

- Do **not** publicly disclose details until a fix is released  
- Do **not** exploit the vulnerability beyond what is required to prove it  
- Do **not** access, modify, or delete data you do not own  

We appreciate responsible researchers who follow coordinated disclosure.

---

## 🛠️ How Vulnerabilities Are Addressed

1. The report is reviewed and validated  
2. A fix is developed on a private branch  
3. A patched release is published  
4. Credit is given (if desired)

Critical vulnerabilities may trigger an out-of-band release.

---

## 🧰 Security Best Practices for Users

To stay safe:

- Always use the **latest version**  
- Keep backups of important Studio One data  
- Review what the Folder Sweeper and other tools will modify  
- Avoid running modified or unofficial builds of StudioOneTools  
- Verify downloaded releases via GitHub’s checksum metadata when available
